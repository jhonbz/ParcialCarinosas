Puerto :8080 — punto de entrada único

Descripción

El API Gateway actúa como puerta de entrada única para todos los microservicios. Recibe las peticiones del frontend y las redirige al servicio correspondiente según la ruta. Implementado con Spring Cloud Gateway.

Rutas de enrutamiento

ROUTE
/api/people/**
Redirige al people-service en puerto :8081
ROUTE
/api/cases/**
Redirige al case-service en puerto :8082
ROUTE
/api/evidence/**
Redirige al evidence-service en puerto :8083
ROUTE
/api/workflow/**
Redirige al workflow-service en puerto :8084

Headers de solicitud recomendados

Content-Typeapplication/json
Acceptapplication/json
Puerto :8081 — gestión de personas

Descripción

Maneja el CRUD de personas (beneficiarios, empleados o involucrados) dentro del sistema Carinosas. Cada persona puede estar asociada a uno o varios casos.

Endpoints

GET
/api/people
Lista todas las personas registradas en el sistema
→ 200 OK · Array de PersonaDTO
GET
/api/people/{id}
Obtiene el detalle de una persona por su ID
@PathVariable id (Long) · → 200 OK | 404 Not Found
POST
/api/people
Crea una nueva persona en el sistema
@RequestBody PersonaDTO · → 201 Created
PUT
/api/people/{id}
Actualiza los datos de una persona existente
@PathVariable id · @RequestBody PersonaDTO · → 200 OK | 404
DELETE
/api/people/{id}
Elimina una persona del sistema por su ID
@PathVariable id · → 204 No Content | 404

Modelo de datos — Persona

idLong
nombreString requerido
apellidoString requerido
documentoString requerido
emailString
telefonoString
fechaRegistroLocalDate
Puerto :8082 — gestión de casos

Descripción

Núcleo del sistema. Gestiona los casos sociales o jurídicos de la organización Carinosas. Cada caso está asociado a una persona y tiene un estado que se controla desde el workflow-service.

Endpoints

GET
/api/cases
Retorna todos los casos registrados
→ 200 OK · Array de CasoDTO
GET
/api/cases/{id}
Obtiene un caso específico por su ID
@PathVariable id · → 200 OK | 404 Not Found
GET
/api/cases/person/{personId}
Filtra todos los casos de una persona concreta
@PathVariable personId · → 200 OK · Array de CasoDTO
POST
/api/cases
Abre un nuevo caso. Se crea con estado inicial ABIERTO
@RequestBody CasoDTO · → 201 Created
PUT
/api/cases/{id}
Actualiza información del caso (descripción, fecha, etc.)
@PathVariable id · @RequestBody CasoDTO · → 200 OK
DELETE
/api/cases/{id}
Elimina (o archiva) un caso del sistema
@PathVariable id · → 204 No Content

Modelo de datos — Caso

idLong
tituloString requerido
descripcionString
personaIdLong (FK) requerido
estadoString (ENUM)
fechaAperturaLocalDate
fechaCierreLocalDate
Puerto :8083 — gestión de evidencias

Descripción

Administra los documentos, archivos y evidencias que soportan cada caso. Una evidencia siempre está ligada a un caseId específico.

Endpoints

GET
/api/evidence
Lista todas las evidencias en el sistema
→ 200 OK · Array de EvidenciaDTO
GET
/api/evidence/{id}
Obtiene una evidencia puntual por su ID
@PathVariable id · → 200 OK | 404
GET
/api/evidence/case/{caseId}
Retorna todas las evidencias asociadas a un caso
@PathVariable caseId · → 200 OK · Array de EvidenciaDTO
POST
/api/evidence
Agrega una nueva evidencia vinculada a un caso
@RequestBody EvidenciaDTO · → 201 Created
PUT
/api/evidence/{id}
Actualiza los metadatos de una evidencia
@PathVariable id · @RequestBody EvidenciaDTO · → 200 OK
DELETE
/api/evidence/{id}
Elimina una evidencia del caso
@PathVariable id · → 204 No Content

Modelo de datos — Evidencia

idLong
nombreString requerido
tipoString (documento, foto, video…)
urlString
caseIdLong (FK) requerido
fechaCargaLocalDateTime
descripcionString
Puerto :8084 — estados y transiciones

Descripción

Controla el ciclo de vida de los casos a través de un motor de estados. Define qué transiciones son válidas y registra el historial de cambios de estado de cada caso.

Estados del ciclo de vida

ABIERTO → EN PROCESO → EN REVISIÓN → CERRADO

Endpoints

GET
/api/workflow
Lista todos los registros de workflow / historial de estados
→ 200 OK · Array de WorkflowDTO
GET
/api/workflow/{id}
Obtiene un registro de transición específico
@PathVariable id · → 200 OK | 404
GET
/api/workflow/case/{caseId}
Devuelve el historial completo de estados de un caso
@PathVariable caseId · → 200 OK · Array de WorkflowDTO
POST
/api/workflow
Registra una nueva transición de estado para un caso
@RequestBody WorkflowDTO · → 201 Created | 400 Transición inválida
PUT
/api/workflow/{id}
Corrige o actualiza un registro de workflow (admin)
@PathVariable id · @RequestBody WorkflowDTO · → 200 OK
DELETE
/api/workflow/{id}
Elimina un registro de transición del historial
@PathVariable id · → 204 No Content

Modelo de datos — Workflow

idLong
caseIdLong (FK) requerido
estadoAnteriorString (ENUM)
estadoNuevoString (ENUM) requerido
motivoString
responsableString
fechaCambioLocalDateTime

Levantar el proyecto

# Desde la raíz del proyecto
docker-compose up --build

# Acceder al sistema
Frontend:    http://localhost:3000
API Gateway: http://localhost:8080

Ejemplo de llamada completa

# 1. Crear una persona
POST http://localhost:8080/api/people
Content-Type: application/json

{
  "nombre": "María",
  "apellido": "González",
  "documento": "1234567890",
  "email": "maria@example.com"
}

# 2. Abrir un caso para esa persona
POST http://localhost:8080/api/cases
{
  "titulo": "Caso de asistencia",
  "personaId": 1,
  "estado": "ABIERTO"
}

# 3. Adjuntar evidencia al caso
POST http://localhost:8080/api/evidence
{
  "nombre": "Formulario inicial",
  "tipo": "documento",
  "caseId": 1
}

# 4. Avanzar el workflow
POST http://localhost:8080/api/workflow
{
  "caseId": 1,
  "estadoAnterior": "ABIERTO",
  "estadoNuevo": "EN PROCESO",
  "motivo": "Revisión iniciada"
}
